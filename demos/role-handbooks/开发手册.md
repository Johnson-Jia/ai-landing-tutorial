# 开发者 · AI 转型操作手册

> 对应教程：[阶段2 全员赋能](../../stage2.html)、[阶段4 闭环试点](../../stage4.html)、[阶段3 基础设施](../../stage3.html)
> 一句话：你是 AI 转型的主力执行者——用 AI 把重复劳动交出去，把精力放到架构、审查、定义问题上。

---

## 一、你的角色定位
AI 时代，开发者的价值从「写代码」转向「**定义问题 + 设计架构 + 审查验证 + 沉淀知识**」。代码大量由 AI 生成，但**你对业务的理解、对架构的判断、对产出的把关，AI 替代不了**。

> 人 + AI，人是根本。AI 扩展你的能力边界，但不替代你的判断。

## 二、核心任务
1. **精通 AI 编码工具**（Claude Code），把重复/模板代码交给 AI
2. **写好 CLAUDE.md / Skill**，把团队规范变成 AI 能读的资产
3. **改代码前先看影响**（代码图谱），避免盲改
4. **审查 AI 产出**，不让错误逻辑溜进去
5. **沉淀可复用能力**（Skill），从「自己快」变「团队快」

## 三、怎么做（按优先级）

### 3.1 精通 Claude Code（决定 2-3 倍产出差距）
- 安装认证 → 四层配置 → 权限（deny 优先）
- **Effort Level**：复杂任务用 high
- **五大扩展组件**：CLAUDE.md（始终注入）/ Slash Commands / Subagents / Skills / MCP
- **上下文三件套**：`/clear`（修正两次后必用）/ `/btw` / `/rewind`
- 详见 [《Claude Code 完全手册》](../../assets/ref/Claude-Code-完全手册.html)

### 3.2 写好 CLAUDE.md（规范资产化）
- 技术栈、提交规范、禁止项写进 CLAUDE.md，提交 Git
- **检验标准**：对每行问「删掉这行，AI 会犯错吗？」不会就删
- 详细指令移进 Skill（按需加载），别堆 CLAUDE.md 烧上下文

### 3.3 改代码前必跑 impact（代码图谱）
- 改任何函数/类前，先跑 `impact` 看爆炸半径
- 提交前必跑 `detect_changes` 看受影响符号
- **铁律**：NEVER edit without impact；NEVER commit without detect_changes

### 3.4 审查 AI 产出（质量把关）
- AI 生成代码 ≠ 正确代码，**必须人审**
- **不让 AI 同时写代码和测试**（会写出验证错误逻辑的测试，全绿但全错）
- 关键规则用 Hook 强制（CLAUDE.md 只是建议，Hook 是确定性的）

### 3.5 沉淀 Skill（从个体到团队）
- 把高频操作做成 Skill 共享（团队 Skill 仓库）
- 写清 description（决定是否自动调用）+ 触发/禁止条件

## 四、你的工具箱
| 工具 | 用途 | 出处 |
|---|---|---|
| Claude Code | 主力 AI 编码 | 阶段2 |
| CLAUDE.md / Skill / Hook | 规范资产化 | 阶段2 |
| 代码图谱（codebase-memory-mcp） | 改代码前看影响 | 阶段3 |
| OpenSpec + Superpowers | 规范驱动闭环 | 阶段4 |

## 五、检查清单
- [ ] Claude Code 已精通（Effort / 扩展组件 / 上下文管理）
- [ ] CLAUDE.md 已提交 Git，每行经「删掉会犯错吗」检验
- [ ] 改代码前必跑 impact，提交前必跑 detect_changes
- [ ] AI 产出必经人审，不让 AI 同时写代码和测试
- [ ] 高频操作已沉淀成 Skill

## 六、常见坑
- ❌ 全员开全权限（`--dangerously-skip-permissions`）当入门配置
- ❌ 跳过探索直接编码（= 重写 bug）
- ❌ 信任 AI 不验证（尤其测试，会被偷改）
- ❌ CLAUDE.md 越写越臃肿（烧上下文）
